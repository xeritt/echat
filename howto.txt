				if (!$user->getPassword()) {

				$params = $request->request->all();

Doctrine
, which integrates tightly with Symfony, supports four types of relationships between entities, defined using PHP attributes (or annotations in older versions). These are: 

    ManyToOne: This is the most common and often considered the "king" of relationships. In a ManyToOne relationship, many entities (e.g., Products) can relate to a single other entity (e.g., a Category). The foreign key is stored on the "many" side (the Product table).
    OneToMany: This is the inverse side of a ManyToOne relationship (e.g., a Category can have many Products). It is mapped by referencing the field on the owning (ManyToOne) side, and typically uses an ArrayCollection to manage the related entities.
    OneToOne: A one-to-one relationship links a single entity to a single other entity (e.g., a User has one Profile).
    ManyToMany: In a ManyToMany relationship, each entity can be related to multiple entities of the other type (e.g., an Article can have many Tags, and each Tag can be associated with many Articles). This is typically managed via a separate join table in the database, but Doctrine abstracts this away, allowing developers to work with objects and collections directly. 

Key Concepts

    Owning vs. Inverse Side: In bidirectional relationships, one side is the "owning" side, and the other is the "inverse" side. The owning side is where Doctrine checks for changes and persists the relationship to the database (it holds the foreign key or manages the join table). The inverse side is mapped by referencing the field on the owning side.
    Mapping: Relationships are defined in entity classes using PHP attributes (like #[ORM\ManyToOne(...)] or #[ORM\ManyToMany(...)]) or annotations (e.g., @ORM\ManyToOne). The targetEntity and mappedBy or inversedBy parameters are used to specify how the entities relate to each other.
    Persisting: To save a relationship, you set the related object(s) on the entity's property and then persist the entity using the EntityManager. Doctrine automatically handles the underlying SQL logic.
    Fetching: Related objects can be fetched automatically when accessed (lazy loading) or explicitly joined in DQL/Query Builder queries to avoid performance issues like the N+1 problem.
